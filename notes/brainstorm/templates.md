# 📜 Grimoire — Brainstorm: Note Templates

> All note templates with full markdown content. Visual conventions, properties, and section structure. Templates ship FLAT in `templates/` regardless of which layer they target; naming carries the role.

## Visual Conventions

- **infobox**: right-floating callout with portrait/image and key data (Fandom Wiki style), activated via `cssclasses: [wiki-entry]` in frontmatter
- **callout types**: `[!quote]` for read-aloud / quoted text, `[!warning]` for creator-only secrets, `[!tip]` for audience-safe knowledge, `[!infobox]` for sidebar data
- CSS snippet shipped with the template for infobox styling (Earliest Usable)

## Template Layout

Templates live FLAT in `templates/` (not mirrored to vault structure). Naming carries the role. Small fixed set, no hierarchy to navigate:

```
templates/
├── character.md             # Layer 1 entity
├── location.md              # Layer 1 entity
├── item.md                  # Layer 1 entity
├── faction.md               # Layer 1 entity
├── lore.md                  # Layer 1 entity (with category: event for canonical timeline)
├── scene.md                 # Layer 2 narrative
├── session-prep.md          # Layer 3 (TTRPG)
├── session-log.md           # Layer 3 (TTRPG)
├── encounter-combat.md      # Layer 3 (TTRPG)
├── encounter-social.md      # Layer 3 (TTRPG)
└── encounter-exploration.md # Layer 3 (TTRPG)
```

No template for `mechanics/` in v1 — that folder is free-form, users populate with system specs, house rules, gameplay design docs as makes sense for their project.

## Properties System

### Common Properties (all entities — Layer 1)

```yaml
---
type: character | location | item | faction | lore
status: draft | ready | revealed | retired
canon: true                  # canonical to the universe? Default true.
source: ""                   # attribution: "Hillenbrand 2018", "Lightbringer Wiki", "self"
tags: []
genai: false                 # is this entity's primary asset AI-generated?
---
```

The `canon: true | false` flag is the key data point for downstream consumers (Remembrance filtering for "canon stories" reads `canon: true` only). Default `true`; users explicitly mark `canon: false` for project-specific inventions that don't fit the source's canon.

### Optional Clue Layer

Any entity can carry a clue role. "Clue" is a role, not a type. Useful for TTRPG mysteries but also for novels and screenplays with mystery elements.

```yaml
clue: true
links-to-secret: "[[secret-name]]"
found-at: "[[location-name]]"
```

### Bases Views (in home.md / views/)

- "All characters" → filter `type: character`, sort by faction
- "Canon only" → filter `canon: true`
- "My inventions" → filter `canon: false`
- "All clues" → filter `clue: true`, columns: file, type, links-to-secret, found-at, status
- "Unrevealed clues" → filter `clue: true` + `status: ready`
- "PCs" → filter `type: character` + `character-type: pc`
- "Generated by AI" → filter `genai: true`
- "Dead / retired" → filter `status: retired`
- TTRPG-only: "All sessions" → filter `type` in `[session-prep, session-log]`
- TTRPG-only: "All encounters" → filter `type: encounter`

## Layer 3 Templates (TTRPG only — used in `play/`)

### Session Prep Template

`templates/session-prep.md`

The "Spoiler Watch" section was intentionally dropped from v1 (the `status` property + callout types already cover the spoiler use case). If a GM finds they want a free-text spoiler list, they can add one to their own session-prep notes.

```markdown
---
type: session-prep
session: 1
date:
planned-scenes: []
status: draft
tags: []
```
